# Project Status

## 文档定位

本文件用于记录 `Personal Digest Agent` 的整体项目进度、阶段目标、运行现状与后续生命周期管理约束。

它不是单纯的需求文档，也不是仅面向某一个阶段的 `MVP status` 文档。  
`MVP` 在这里仅表示项目已经完成过的一个阶段性里程碑，而不是项目的最终目标。

## 当前运行现状

### 1. 真实邮件发送

已经验证成功的发送模式：

- `smtp.gmail.com`
- `port = 465`
- `use_ssl = true`
- `use_tls = false`

### 2. 当前配置中存在的临时项

以下配置属于验证阶段临时调整，不应默认视为最终长期配置：

- `config/sources.yaml` 中的 `min_score = 40`
  - 该值是为了让此前降级分析得到 `score=40` 的文章进入 Digest
  - 后续应重新决定是否恢复到 `60` 或其他正式值

- `initial_fetch_entry_limit`
  - 当前已加入首轮抓取上限机制
  - 但具体默认值是否长期保持，仍需要结合真实使用情况确认

### 3. 敏感配置管理

项目中的敏感配置不应以明文形式保存在仓库文件中。

当前推荐做法：

- `config/settings.yaml` 只保留环境变量占位符
- 敏感值通过运行环境注入
- 多收件人通过 `SMTP_RECIPIENTS` 以逗号分隔提供

需要外置的配置包括：

- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`
- `SMTP_USERNAME`
- `SMTP_PASSWORD`
- `SMTP_SENDER`
- `SMTP_RECIPIENTS`

如果这些值曾进入版本历史，应尽快执行密钥或密码轮换。

### 4. 当前数据状态

该项目当前已跑通过真实链路，但数据库中的历史数据仍有收敛空间。

需要持续关注的不是“能不能跑”，而是“历史数据如何管理、如何保持干净、如何避免半成品长期堆积”。

## 当前主要问题

### 1. 历史数据策略不完整

当前最核心的问题不是功能缺失，而是数据生命周期管理不完整：

- 首轮抓取、手动中断和临时验证都可能留下中间态数据
- 需要明确：
  - `pending_extraction` 何时继续处理
  - 哪些文章应该直接丢弃
  - 哪些失败文章应该重试
  - 哪些旧分析结果应该重算

### 2. 运维操作仍不够明确

目前主链路可用，但日常维护仍偏手工，缺少明确命令来处理：

- 查看当前状态
- 清理失败数据
- 重试提取
- 重试分析
- 重新发送指定日期 Digest

### 3. 邮件内容质量仍处于第一版

当前邮件已经能送达，但在“长期阅读体验”层面仍有明显提升空间：

- 分类组织方式
- 摘要长度一致性
- 标签展示方式
- 标题和来源信息排版
- 内容密度控制

### 4. 默认配置尚未完全定稿

当前项目已可用，但一些默认值仍然是“验证可行性后的临时结果”，尚未形成长期标准。
